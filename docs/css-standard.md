## 0. 실측 현황 (이 설계의 근거)

도현 실측 + 나루 재측정 (`src/` 전체, `*.tsx` 기준):

| 항목 | 현황 | 증거 |
|---|---|---|
| 페이지 컨테이너 너비 | 6xl / 2xl / 7xl / 6xl 제각각 (672~1280px) | page.tsx:209, terms:8, privacy:8, shared/[id]:64, about:60 |
| 좌우 패딩 | `px-4 sm:px-6` / `px-5` / `px-6` 혼재 | 위 동일 |
| raw 색상 출현 | **916회** | `(slate\|gray\|blue\|amber\|...)-[0-9]` grep |
| 시맨틱 토큰 출현 | **145회** | `(bg\|text\|border)-(background\|foreground\|muted\|...)` grep |
| raw 색상 집중도 | 상위 6파일이 660회(72%). PrdViewer 혼자 273회 | 파일별 카운트 |
| shadcn 토큰 | **이미 풀세트 정의됨**(globals.css :68~135), base는 무채색 neutral | globals.css |
| Tailwind | **v4 CSS-first**. `tailwind.config` 없음, `@theme inline` 사용 | globals.css:1, postcss.config.mjs |

핵심 판단 3가지:
1. **토큰은 새로 만들 게 없다.** globals.css에 shadcn 토큰 다 있다. 우리가 할 건 "raw를 토큰으로 바꾸는 이행"뿐.
2. **base가 무채색(neutral)이라** `slate-*` → `muted-foreground` 같은 치환이 **눈에 거의 안 띈다.** 회귀 위험이 낮다는 뜻. 이행하기 좋은 조건.
3. Tailwind v4라 **config로 스케일을 강제할 수 없다.** 스케일 합의는 "컨테이너 컴포넌트 + 이 문서의 규약"으로 지킨다. 린트로 막는 게 아니라 리뷰(세린)로 막는다.

## A. 레이아웃 컨테이너 컴포넌트

### 결정: 컴포넌트 1개 `<PageContainer>`

**위치:** `src/components/layout/PageContainer.tsx`
(`layout/` 디렉토리 신설. 앞으로 레이아웃성 컴포넌트는 여기로.)

**너비는 정확히 3단.** 그 이상 안 만든다.

| width 값 | max-w | px | 용도 |
|---|---|---|---|
| `narrow` | `max-w-2xl` (672px) | 폼·약관·개인정보·로그인 등 **읽기/입력 단일 컬럼** |
| `default` | `max-w-6xl` (1152px) | **앱 본문 기본**. 메인 워크플로우 페이지 |
| `wide` | `max-w-7xl` (1280px) | 문서 뷰어처럼 **좌측 트리+우측 본문** 넓은 작업 화면 |

> 5xl/4xl/3xl/lg/md 등은 **컨테이너 레벨에서 금지**. (about 랜딩 내부의 `max-w-3xl` 같은 건 "문단 readability용 inner 제약"이라 예외 — 아래 매핑 참고.)

**좌우 패딩 단일화:** `px-4 sm:px-6` 하나로 통일.
- 모바일 16px, sm 이상 24px. 가장 흔히 쓰이던 page.tsx 값을 표준으로 채택(이미 그게 기본이라 회귀 최소).
- `px-5`(terms/privacy), `px-6`(about) 는 폐기 → 표준으로 흡수.

**세로 리듬(섹션 간격) 처리:** props로 강제하지 않는다. 컨테이너는 **외곽 세로 패딩만** 책임진다.
- 기본 `py-6 sm:py-8` (앱 본문), 약관류는 `py` prop으로 `py-10` 허용.
- 섹션 사이 간격은 컨테이너 안에서 `space-y-*`로 — 단, **B의 스케일만** 쓴다.

### props 인터페이스 (확정)

```ts
// src/components/layout/PageContainer.tsx
type ContainerWidth = 'narrow' | 'default' | 'wide';

interface PageContainerProps {
  width?: ContainerWidth;   // 기본 'default'
  className?: string;        // 추가 조정용 (배경/세로패딩 override)
  children: React.ReactNode;
}

const WIDTH_MAP: Record<ContainerWidth, string> = {
  narrow:  'max-w-2xl',
  default: 'max-w-6xl',
  wide:    'max-w-7xl',
};

// 렌더: <div className={cn('mx-auto w-full px-4 sm:px-6', WIDTH_MAP[width], className)}>
```

- **`mx-auto px-4 sm:px-6`는 컴포넌트가 박는다.** 호출부에서 못 바꾼다(그게 표준화의 핵심).
- 배경 그라데이션(`min-h-screen bg-gradient...`)은 컨테이너 책임 아님. **페이지 바깥 래퍼**가 갖는다. 컨테이너는 폭/패딩만.
- `cn`은 기존 `src/lib/utils.ts`(shadcn 기본)에 있을 것 — 없으면 태오가 확인.

### 페이지 → 너비 매핑표 (확정)

| 페이지 | 현재 | → 표준 width | 비고 |
|---|---|---|---|
| `app/page.tsx` (메인) | max-w-6xl | **default** | 그대로. px만 표준 확인 |
| `app/terms/page.tsx` | max-w-2xl, px-5, py-10 | **narrow** + `py-10` | px-5→표준 |
| `app/privacy/page.tsx` | max-w-2xl, px-5, py-10 | **narrow** + `py-10` | px-5→표준 |
| `app/shared/[id]/page.tsx` | max-w-7xl(배너) | **wide** | 본문은 PrdViewer. 배너도 wide로 폭 일치 |
| `app/about/page.tsx` | max-w-6xl 반복 | **랜딩 예외 — 적용 보류** | 아래 |
| `AuthGate`(로그인 폼) | max-w-sm/md | **narrow 또는 그대로** | 카드형 폼은 좁은 게 정상. 무리해서 안 바꿈 |

**about(랜딩)은 이번 표준화 대상에서 제외한다.** 이유:
- 마케팅 랜딩은 섹션마다 의도적으로 폭/리듬이 다르다(hero `max-w-3xl`, 본문 `max-w-2xl`, 그리드 `max-w-6xl`). 앱 본문 규칙을 억지로 끼우면 오히려 망가진다.
- 단, **바깥 섹션 폭은 max-w-6xl로 이미 일관**돼 있다. 굳이 손대서 회귀 만들 이유 없다.
- 정 통일하고 싶으면 그건 별도 "랜딩 리디자인" 과제로. 이 정의서 범위 밖.

## B. spacing 스케일 (허용 단 좁히기)

현황: gap/space-y 2·1·3·4·6·10·12 산발. → **6단으로 좁힌다.** 4px 그리드.

| 토큰 | 값 | 쓰는 곳 |
|---|---|---|
| `1` (4px) | 아이콘-텍스트 등 **밀착** | 인라인 미세 간격 |
| `2` (8px) | 버튼 그룹, 칩, 폼 라벨-입력 |
| `3` (12px) | 카드 내부 요소 사이 |
| `4` (16px) | **기본 블록 간격** (가장 많이) |
| `6` (24px) | 카드/패널 사이, 섹션 내 그룹 |
| `8` (32px) | 페이지 섹션 사이 (앱 본문) |

- **금지: 5, 7, 9, 10, 12 직접 사용.** (랜딩 `py-24` 등 마케팅 큰 여백은 about 예외라 제외.)
- 반응형은 `gap-3 sm:gap-4` 처럼 **인접 단끼리만** 점프. 2→6 같은 큰 점프 금지.
- 강제 수단은 린트 아님(Tailwind v4라 못 막음) → **리뷰 체크리스트**(E 참고).

## C. 타이포 스케일 (헤딩 단계)

현황: 헤딩 xl~6xl 혼재. → **앱 본문은 4단으로 고정.** (랜딩 6xl은 about 예외)

| 역할 | 클래스 | 용도 |
|---|---|---|
| `page-title` | `text-2xl sm:text-3xl font-bold tracking-tight` | 페이지 H1 (1개만) |
| `section` | `text-lg sm:text-xl font-semibold` | 섹션 제목 H2 |
| `subsection` | `text-base font-semibold` | 카드/하위 제목 H3 |
| `body` | `text-sm` (보조 `text-xs`) | 본문/설명 |

- 메인 page.tsx의 `text-2xl sm:text-3xl md:text-4xl`는 **md:text-4xl 빼고** 위 page-title로 정렬(3단→2단 반응형으로 단순화).
- 본문 기본은 `text-sm`. 화면 정보밀도가 높은 앱이라 base 16px(`text-base`)를 본문 전체에 쓰면 오히려 헐렁해 보인다 — 현재도 sm 위주라 회귀 적음.
- 색은 타이포에 박지 말 것. 제목은 `text-foreground`, 보조는 `text-muted-foreground` (D 참고).

## D. 색상 점진 이행 가드레일

### 절대선: **916개 한 방 치환 금지.**

이유: PrdViewer(273), ScreenDiagram(87), WBS(83), TestPlan(82) 같은 시각화 컴포넌트는 색이 **의미를 가진다**(상태색, 다이어그램 노드색). sed로 일괄 치환하면 의미색까지 뭉개진다. 자동 치환은 회귀 폭탄.

### 치환 사전 (raw → 토큰) — base가 neutral이라 시각 변화 거의 없음

| raw 패턴 | → 토큰 |
|---|---|
| `text-slate-900 / -800 / dark:text-slate-50` | `text-foreground` |
| `text-slate-600 / -500 / -400` | `text-muted-foreground` |
| `bg-slate-50 / -100 / white` (표면) | `bg-card` 또는 `bg-background` |
| `bg-slate-100 / -200` (옅은 면) | `bg-muted` |
| `border-slate-200 / -300` | `border-border` |
| `bg-blue-600 text-white` (주요 버튼) | `bg-primary text-primary-foreground` → 가능하면 `<Button>` 컴포넌트로 |
| `text-blue-600 underline` (링크) | `text-primary underline` |
| `bg-red-* / text-red-*` (오류) | `text-destructive` / `bg-destructive` |

### 의미색은 토큰화하지 않고 **유지**한다 (이행 대상 아님)

- 다이어그램 노드색, 상태 배지(진행/완료/오류를 색으로 구분), 랜딩 그라데이션 — 이건 raw 색 그대로 둔다. 토큰 6~7개로는 표현 못 한다.
- 단 **오류=destructive, 성공=의미 유지**처럼 명백한 건 위 사전 적용.

### 신규/수정 코드 규칙 (앞으로)

1. **새로 쓰는 코드는 무조건 토큰.** raw 색 신규 추가 금지(의미색 제외).
2. 기존 파일을 **다른 이유로 만질 때** 그 파일의 raw 색을 같이 토큰화(보이스카웃 룰). 별도 색만 고치는 대량 PR은 안 만든다.
3. 버튼/입력/배지는 **raw로 새로 만들지 말고 `src/components/ui/*` 사용.**

## F. PrdViewer 본문 레이아웃 (우측 본문 한정)

> 오너 불만: "문서 탭 우측 본문이 정돈 안 됐다." 도현 실측(PrdViewer.tsx 2544줄, 본문 1784~)으로 4가지 손본다.
> **범위 못박음: "우측 본문의 카드·세로간격·버튼·빈공간"만.** 색 대량치환·2544줄 분해·뷰모드 로직·트리 사이드바는 **이번 아님.**
> D절 보이스카웃선과 충돌 안 남: 여기 허용은 **레이아웃(간격/카드/버튼배치)뿐**, 색은 의미색 보존(빨강=오류·노랑=경고 유지).

### F-1. 세로 리듬 — 배너 3종을 본문 space-y 안으로

- 본문 카드 묶음(1933 `<div className="space-y-4">`)은 이미 **`space-y-4`(B의 `4`)** 정상. 이 값 유지·확정.
- 문제는 **그 밖에 떠 있는 배너 3종**(1787·1818·1886)이 각자 `mb-4`로 컨테이너(1784) 직속에 붙어 있는 것. 들쭉날쭉의 근원.
- 지시: 1784 컨테이너를 **`space-y-4` 컨테이너로** 만들고(또는 배너 3종+1922 본문 블록을 하나의 `space-y-4` div로 감싸고), 배너 3종의 **`mb-4`를 전부 제거.** 세로간격은 컨테이너 `space-y-4` 단일 출처가 책임.
- **금지: 배너마다 mb-* 산발.** B 스케일 밖(`mb-3`/`mb-2` 등) 신규 추가 금지.

### F-2. CardContent 패딩 — 세로축만 `py-4 sm:py-6`로 통일

좌우는 이미 `px-2 sm:px-6`로 정렬돼 있다(전 카드 공통). **세로축만 갈린다.** 하나로:

| 위치 | 현재 | → 통일 |
|---|---|---|
| 2160 (편집) | `px-2 sm:px-6 pt-4 sm:pt-6` | `px-2 sm:px-6 py-4 sm:py-6` (pt→py) |
| 2190 (시각화) | `px-2 sm:px-6 py-4 sm:py-6` | 그대로 (= 표준) |
| 2296·2388 (미리보기·raw) | `px-2 sm:px-6 py-4 sm:py-6` | 그대로 |

- **표준값 = `px-2 sm:px-6 py-4 sm:py-6`.** (이미 다수가 이 값 → 회귀 최소. 1784 컨테이너 패딩과도 동일해 일관.)
- **예외 1개 허용: 2398 빈 상태 카드**(`text-center px-3 sm:px-6 py-12 sm:py-16`). 문서 미생성 시 "생성하기" CTA를 세로 중앙에 띄우는 의도적 큰 여백 → **유지.** 단 `px-3`→`px-2`로만 맞춰 좌우 통일(세로 `py-12 sm:py-16`은 빈상태 예외).

### F-3. 배너 3종(1787·1818·1886) → `<Card>`화, 의미색은 보존

raw `rounded-lg border ... p-3` div를 `<Card>`로 교체. **모양만 Card에 얹고 색은 className으로 유지.** `src/components/ui/card.tsx` 이미 있음.

- 방식: `<Card className="border-red-200 bg-red-50 dark:border-red-700 dark:bg-red-900/30 p-3">` 형태. **Card 기본 border/배경을 의미색 className으로 덮어쓴다.** 내부 텍스트·아이콘 색(`text-red-*`)은 그대로.
  - 1787 빨강(다이어그램 깨짐=오류) → 빨강 유지
  - 1818 노랑(영향받은 하위문서=경고) → 노랑 유지
  - 1886 회색/파랑(상위 모순 안내=정보) → 그대로 유지
- **`border-destructive` 같은 토큰 치환 쓰지 말 것.** 노랑·정보색은 토큰에 없다(D절: 의미색은 토큰화 안 함). 빨강도 여기선 토큰 대신 raw 유지 — 이 4블록은 색 이행 대상 아님(D 5단계 보류선).
- CardContent 안 쓰고 `<Card>`에 직접 `p-3` 둬도 됨(배너는 헤더/콘텐츠 분리 불필요). **과설계 금지.**

### F-4. 버튼 그룹핑(2015~) — 시각적 묶음까지만

문서 헤더 우측 버튼 6개(복사·보기모드·인쇄·편집·내보내기·다시생성)가 `flex flex-wrap gap-2` 한 줄로 늘어섬.

- 지시(단순한 쪽): **아이콘 보조액션(복사·인쇄)을 `border rounded-md` 한 덩어리로 묶는다.** `<div className="inline-flex rounded-md border divide-x">` 안에 복사/인쇄 아이콘 버튼(`variant="ghost"`)을 넣어 한 묶음으로. 나머지(편집·내보내기·다시생성)는 개별 유지.
- 바깥은 `flex flex-wrap gap-2` 유지. **그룹 1개만 만들고 끝.** 드롭다운/툴바 컴포넌트 신설 금지(과설계).
- **중복 지적:** "다시생성"이 **1832 영향배너에도** 있음(`순서대로 모두 갱신`). 헤더의 다시생성과 의미 겹침. 이번엔 **건드리지 말고 기록만** — 배너의 건 "영향받은 N개 일괄", 헤더의 건 "현재 문서 1개"라 맥락이 달라 둘 다 필요. 추후 라벨로 구분("이 문서 다시생성" vs "영향문서 갱신") 검토. 이번 범위 밖.

### F-5. 시각화 기본 진입을 preview로 (빈 패널 제거)

### F-7. 좌우 인셋 단일책임 (계단 끝선 제거)

> 오너 재불만: "문서 탭 우측 본문이 아직도 울퉁불퉁." 도현 정밀진단으로 원인 확정.
> F-2는 **CardContent 안쪽 패딩과 세로축만** 통일했다. 그래서 ①헤더바·②컨텐츠래퍼·③배너·④카드가 **서로 다른 좌우 기준선을 갖는 구조 자체**는 안 건드렸다 → 우측 본문 블록들의 좌측 시작선이 16/24/40/48px로 제각각이라 계단.
> **이번도 색 안 만진다. 좌우 정렬 클래스(px/pl/mx)만.** 세로축(py/space-y)·뷰모드로직·273색·2544줄분해·트리는 안 건드림.

- **⑤ 카드 내부 헤더/콘텐츠 불일치** — shadcn `card.tsx`는 CardHeader·CardContent 둘 다 기본 `px-4`(16px). 그런데 PrdViewer가 CardContent만 `px-2 sm:px-6`로 **덮어써서**(8/24px) 같은 카드 안에서 제목줄과 본문줄의 좌측이 어긋난다(데스크탑 16 vs 24). F-2가 이 오버라이드를 "통일"이라 두고 갔지만 사실 헤더와는 안 맞았다.

#### 원칙 — "좌우 인셋은 한 군데가 책임진다" + "박스/비박스 구분"

1. **좌우 패딩 단일 출처 = 컨텐츠 래퍼(1797).** 이 래퍼 하나가 "본문 영역의 좌우 끝선"을 정한다.
2. **래퍼의 직계 비(非)박스 블록(배너·네비·완료표시)은 좌우 패딩/마진 0.** 끝선은 래퍼가 책임. 자기가 px를 더하면 계단난다.
3. **카드(시각적 박스)는 안쪽 패딩을 가져도 된다 — 단 모든 카드가 동일값.** 카드는 테두리 있는 박스라 안쪽 여백이 정상이고, 카드끼리 같은 안쪽 패딩이면 박스 바깥 좌측선은 래퍼 기준으로 자동 일치한다. **카드 안쪽은 헤더=콘텐츠 동일**해야 한다.

> 한 줄 규칙: **"박스가 아니면 좌우는 래퍼가 책임(자식 px 0). 박스면 카드 기본 패딩에 맡기되 전 카드 동일."**

#### 확정 — 좌우 단일값

**래퍼(1797)의 좌우 패딩을 데스크탑/모바일 모두 표준으로 맞춘다.**
- 현재 `px-2 sm:px-6`(모바일 8 / 데스크탑 24) → **`px-4 sm:px-6`**(모바일 16 / 데스크탑 24)로.
- 이 값은 A절·PageContainer 표준(`px-4 sm:px-6`)과 **동일**. 본문 좌우 끝선이 앱 전체 페이지 표준과 같아진다.
- 세로(`py-4 sm:py-6 space-y-4`)는 **그대로 유지**(F-1·F-2 결정 보존).

> CardContent에서 `px-*`를 떼면 카드 기본 `px-4`(16px)로 돌아간다. 데스크탑 본문 텍스트 시작선이 24→16px로 **8px 왼쪽 이동**한다(카드 바깥 래퍼24 + 카드안16 = 40px, 전엔 48px). 이건 의도된 결과 — **CardHeader(16)와 같아지는 게 목적**이라 카드 안에서 제목·본문이 비로소 한 줄로 선다.

#### 헤더바(①) 정렬 — sticky 폭전체 특성 고려

헤더바는 본문 래퍼와 **형제(둘 다 1564 `flex-1` 안)**이고, sticky로 폭 전체를 쓴다. 본문 끝선과 맞추되 sticky 특성은 보존.

- **데스크탑**: 헤더바 `sm:px-6`(24) = 래퍼 `sm:px-6`(24). **이미 일치 — 손 안 댐.**
- **모바일**: 헤더바는 `pl-16`(64, 햄버거 버튼 자리) + 우측 `px-4`(16). 래퍼는 `px-4`(16, 위에서 8→16 변경 후). **우측(16)은 일치.** 좌측은 `pl-16`이 **햄버거 플로팅 버튼을 피하려는 의도적 오프셋**이라 본문과 다른 게 정상(헤더바 텍스트가 버튼에 안 깔리게). → **헤더바 모바일 pl-16 유지.**
- 결론: **헤더바(1567)는 변경 없음.** 데스크탑은 이미 24로 본문과 일치, 모바일 좌측 차이는 햄버거 회피라 의도적. 래퍼를 8→16으로 올리면 모바일 우측선도 헤더바와 맞아 떨어진다.

> pl-16 함정 주의: 모바일에서 헤더바 좌측만 64px인 건 "어긋남"이 아니라 햄버거 버튼 영역. **여기에 본문 래퍼를 맞추려고 pl-16을 본문에 넣지 말 것**(본문은 햄버거가 떠 있는 영역이 아님 — 헤더바에만 버튼 있음).

#### F-7 태오 적용 순서 (회귀 낮은 순)

> 각 단계 독립 커밋. 단계마다 빌드 + 문서탭 **데스크탑·모바일 둘 다** 육안 확인.

1. **[정정] 완료배너 1988 `px-4`→`px-3`**(`px-4 py-2`→`px-3 py-2`). **박스라 좌우 px 제거가 아니라 px-3으로 통일**(배너3종 p-3과 안쪽값 일치). 만약 태오가 이미 `py-2`(제거)로 적용했다면 **`px-3 py-2`로 되돌릴 것**.
2. **CardContent 좌우 px 제거**(2176·2206·2312·2404·2414, 그리고 빈상태 2414는 px-2만 제거하고 py-12 sm:py-16 유지). 카드 안 헤더=콘텐츠 정렬. 데스크탑 본문이 8px 왼쪽으로(의도). **2272·2299 p-0/p-3은 제외.**
3. **래퍼 1797 `px-2`→`px-4`**(모바일만 8→16, 데스크탑 sm:px-6 불변). 모바일에서 배너·카드 좌우가 헤더바 우측선과 맞음. 데스크탑 영향 0(이미 24).

- 헤더바(1567)는 **이 과제에서 변경 없음**(위 정렬 분석대로).
- 각 단계는 **데스크탑/모바일 영향 분리**돼 있다: 1은 양쪽, 2는 주로 데스크탑(카드안 8px 이동), 3은 모바일만(8→16).
