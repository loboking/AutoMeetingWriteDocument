## 한 줄 정의
**회의 녹음 한 번으로 PRD·화면설계·DB·API명세 등 개발 기획 문서 14종을 자동 생성하는 웹 SaaS.**

## 누구의 문제인가 (타겟)

베타에서는 넓게 안 잡는다. **"문서를 만들어야 하는데 만들 사람이 없는"** 한 부류에 집중.

| 우선 | 타겟 | 이들의 상황 |
|------|------|-------------|
| **P0 (핵심)** | 기획 인력 없는 **소규모 개발팀 / 1인 개발자 · 외주개발사 대표** | 회의는 했는데 PRD·산출물을 직접 써야 함. 문서 쓰는 시간이 곧 개발 못 하는 시간 |
| **P1** | **SI·에이전시의 주니어 기획자/PM** | 산출물 종류가 많고(14종) 포맷(Word/Excel/PPT)까지 맞춰야 함. 반복·정형 작업에 며칠 소모 |
| P2 (나중) | 사내 신규 프로젝트 발의자(비개발 직군) | 아이디어 회의는 했는데 개발팀에 넘길 문서가 없음 |

> 베타 검증 대상은 **P0**. "기획자가 아예 없어서 문서가 안 만들어지는" 고통이 가장 크고 명확하다.

## 무슨 문제 (Problem)
> **"회의는 끝났는데, 문서는 아무도 안 만든다."**

- 회의 결정사항이 문서화 안 됨 → 개발 단계에서 누락·재논의
- PRD/화면/DB/API 산출물 작성에 기획자 **며칠** 소요 (반복·정형 작업)
- 소규모 팀·1인 개발자는 **기획 인력 자체가 부재** → 문서 없이 개발 강행 → 품질·일정 리스크
- 기존 회의록 도구는 **요약까지만**. 개발에 바로 쓰는 산출물로 안 이어짐

## 어떻게 (How / 처리 흐름)
1. **입력** — 브라우저 녹음 또는 녹음 파일 업로드
2. **회의록** — STT로 텍스트화 → AI가 개요·논의·결정·할일로 구조화
3. **문서 생성** — 요약 기반으로 14종을 의존성 순서대로 자동 작성 (상위→하위 전파)
4. **활용** — Markdown·Word·Excel·PowerPoint로 내보내 팀 공유

## 무엇으로 (차별점 / Why Us)

| 구분 | 일반 회의록 도구 (Otter, Tldv, 클로바노트 등) | MeetingAutoDocs |
|------|------|------|
| 산출물 | 회의 **요약까지** | **개발 실무 문서 14종** |
| 문서 연계 | 없음 | **문서 간 의존성 자동 전파** (PRD 바뀌면 하위 반영) |
| 내보내기 | 텍스트 | **Word·Excel·PowerPoint·MD** (제출용 포맷) |
| 시각화 | 없음 | **플로우차트·ERD·화면 다이어그램 내장** |

> 한 문장 차별화: **"회의록이 아니라, 회의에서 개발 문서까지."** 경쟁은 요약에서 멈추고, 우리는 산출물에서 시작한다.

## 핵심 가치 (한 줄)
- **고통 제거형 가치**: "기획자 며칠"을 "회의 직후 자동"으로. 없던 문서를 있게.
- 절약형이 아니라 **존재 가능형**(P0에겐 안 만들어지던 게 만들어진다)이라 체감이 크다.

## 베타에서 검증할 가설 (도윤·다음 PM 사이클 입력)
| # | 가설 | 검증 신호 |
|---|------|-----------|
| H1 | P0(기획자 없는 팀)이 가장 강하게 반응한다 | 가입자 중 1인/소규모 비중, 재방문률 |
| H2 | 생성된 14종을 **실제로 다 쓰지 않고 2~4종만 쓴다** | 문서별 열람·내보내기·편집 빈도 (→ 요금제 축에 직결) |
| H3 | "회의 1건 = 문서 1세트" 가 핵심 사용 단위다 | 월 회의 처리 건수 분포 |
| H4 | 내보내기(Word/Excel/PPT)가 결정적 가치다 | 내보내기 실행률 |
| H5 | STT 품질·문서 품질이 이탈 1차 원인이다 | 생성 후 재생성/이탈 로그 |

> **H2·H3는 요금제 축의 근거가 된다** → `pricing-axis.md` 참조.
